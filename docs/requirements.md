# 要求定義

## 前提

- 実験データ解析一般のベストプラクティス視点で記述する
- 手段（DB、ファイル、特定ツール等）は問わない
- フレームワークが管理するデータは、意味的な識別子の組み合わせ（被験者、条件、計測種別、処理段階等）で一意に特定可能であることを前提とする。データ値の型（数値、テキスト、バイナリへのパス等）は問わないが、意味的座標を持たないデータはフレームワークの管理対象外である
- データがフレームワークの管理下にあるか否かは明示的な宣言行為（ステージ定義等）によって決まる。管理外のデータ（探索的なnotebook出力等）に対してフレームワークは関与しない
- I〜Vはフレームワークとしてのコア要求。VIは主要利用者の日常ワークフローに関する要求であり、フレームワークの設計はVIに依存しない。ただしVIを自然に支えるIF提供はコア責務に含まれる（→I-6）

## 設計原則

**暗黙依存の外部化**: システムの価値は、解析者の頭の中にある暗黙的な依存（データの意味、パラメータの経緯、処理の前提条件等）をどれだけ明示的に外部化できるかで測られる。その外部化を自然に促す構造であること。

**構造的保証**: 管理下のデータの属性構造（識別軸とその語彙）は、実データに先行して宣言的に定義される。定義に準拠しないデータは管理下に存在できない。識別軸自身も構造化されたメタデータを持ちうる（被験者IDに身長・体重、データ種別IDに単位・意味等）。「暗黙依存の外部化」がなぜ必要かを示す動機であるのに対し、構造的保証はその技術的実現原則である。

**データ中心**: 常に求められているのはデータであり、解析はそのデータを生んだ手段である。因果の方向は「このデータはなぜこの形で存在するか」→「この解析がこのパラメータで適用されたから」。意味・経緯・パラメータはデータの属性としてデータから到達可能。

**不変性と純粋関数性**: 処理ステップは `f(入力データ, パラメータ) → 新データ` という純粋関数。入力データを変更しない。過去の成果物（状態）に影響を与えてはいけない。既存データへの加工は常に新データの生成である。

**データ依存DAG**: すべてのデータ間関係は有向非巡回グラフ（DAG）を形成する。ノードはデータ、辺は処理ステップ。パラメータは辺に付随し、変更時は下流ノードが無効化される。各ノードは有効性状態を持つ。

設計の精神は「明日自分の席に別の人が座るとして、今までの結果をそのまま使えるか」「ある辺が上流ノードのどの部分を使っているかを、コードを読まずに把握できるか」に集約される。

## 要求リスト

### I. アクセス（このデータはどこ？）

**I-1. 単一アクセスポイント**: 解析者が知るべき入口は1つ。そこからスライス・参照・一覧のすべてが行える。横断比較・結合に必要なデータ取得もこの入口経由で完結する。また、外部DAGからのデータ取得IFとしても機能する（→IV-2）。

**I-2. 多軸スライス**: 任意の属性の組み合わせでデータを絞り込める。属性の例：被験者、条件、計測種別、身体部位、処理段階。絞り込みの順序に依存しない。処理段階も属性の一つとして扱う（中間産物へのアクセスも同じインターフェース）。ノードの属性値は単一値または集合をとりうる（全被験者のデータと特定被験者のデータが同一の操作でヒットする）。識別軸自身が構造化された属性を持ちうる（被験者IDに身長・体重、データ種別IDに単位・意味等）。識別子の属性もスライスやフィルタの対象となる。

**I-3. 一意な参照**: ある特定のデータ片を、曖昧さなく指し示せる。「被験者1・条件A・左膝・角度・正規化済み」と言えば1つに定まる。一意性は意味的座標（識別子の組み合わせ）によって成立する。物理パスやIDではなく、意味的座標がデータの正規の参照方法である。

**I-4. DAGに紐づいた一覧性**: 「今、何のデータが存在し、DAG上のどこにいて、有効性状態はどうか」を網羅的に確認できる。属性単位の一覧に加え、DAG構造上の位置と状態が可視であること。外部の利用者が「何があるか」を把握するためにも必要（→IV-2）。

**I-5. 異質なデータ構造の共存**: 時間軸・サンプリングレート・次元が異なるデータ（時系列、スカラー、メタデータ）が、それぞれの構造を保ったまま共存できる。単一構造への強制的な正規化を要求しない。

**I-6. 管理境界IF**: 管理外のデータを管理下に昇格させるIF、および管理下データを管理外環境から利用するIFを提供する。管理下への参入は宣言的行為であり、その手段の提供がフレームワークの責務。管理外環境からの消費と流入の両方をカバーする。

### II. 来歴・整合性（このデータはなぜこう？）

**II-1. DAG整合性**: DAGの全辺が現在のパラメータと整合している。各ノードのデータが、それを生成した純粋関数の現在のパラメータで実際に生成されたものであることが保証される。不整合が検知または防止される。

**II-2. スナップショット再現性**: 同一の生データ + 同一のパラメータスナップショットがあれば、同一の出力を再現できる。

**II-3. ノードの有効性管理**: 各ノードは有効/無効/未生成の状態を持つ。ノードの無効化（データ破棄・パラメータ変更等）時に下流ノードが連鎖的に無効化される。無効ノードが同一出力IFで再生成されれば、下流も復活可能。探索は「未生成・無効状態のノードを作る」こと、確定は「有効状態にする」こと。

**II-4. 部分再生成**: パラメータ変更時に、DAG上の影響を受ける部分木のノードのみが再生成される。全データを最初から作り直す必要がない。

**II-5. パラメータの追跡可能性**: あるデータ片に影響したパラメータが何であるか、データから追跡できる。外部DAGから取り込んだノードについても、出典元DAGへの参照を通じて来歴が追跡可能であること（→IV-2）。

### III. 格納（このデータはどう保持される？）

**III-1. 格納の中立性**: 格納形式が、実行可能な解析手法や格納可能なデータの種類を制約しない。フレームワークはデータ値の型を制約しないが、意味的座標への紐付けは要求する。

**III-2. 拡張性**: 新しい指標・新しい処理段階を追加したとき、既存データへのアクセス方法が壊れない。新しい識別軸の追加や、既存識別子への属性追加も含む。

**III-3. データからの意味到達性**: データのキーや値が何を意味し、どういう意図で作られたかに、データを起点として到達できる。帰属単位はDAGのノード。

- commit履歴は暗黙的な来歴を提供するが、到達性が低い。明示的ドキュメントが履歴を補完する構造
- ドキュメントはデータと同一の管理単位に同居する
- 意思決定履歴が追跡可能である

### IV. 進化と合成（DAGの時間的進化と空間的合成）

**IV-1. スナップショット分岐**: 同じデータに対して複数の解析パスを並行して探索できる。DAGの時間的進化を管理できること。

**IV-2. DAG間の合成可能性**: 別のリポジトリの任意のノードを、新たなDAGの入力ノードとして取り込める。データ実体のコピーによる再現性確保と、出典元への参照保持による来歴追跡性を両立する。

IV-2は複数の基本要求が「外部の利用者にも成立する」ことを求める横断的な要求である:

- I-1: アクセスポイントが外部向けIFとしても機能する
- I-4: 外部利用者が把握可能な一覧性
- II-5: DAGをまたぐ来歴追跡
- III-3: 外部利用者にとっての意味理解可能性

### V. 再利用性（望ましい要求: DAGの辺の設計品質）

**V-1. 解析プロセスの切り出し**: 解析処理を再利用可能な単位で切り出せる。

**V-2. 解析と出力の分離**: 解析ロジックと出力表現（グラフスタイル等）が分離され、差し替え可能。出力パラメータは解析ではなく成果物に帰属する。

### VI. ワークフロー（個人的な利用形態の要求）

フレームワークのコア要求ではないが、主要利用者の日常ワークフローとして記載する。フレームワーク設計はこのカテゴリに依存しない。

**VI-1. 探索的分析ワークフロー**: 管理外環境（notebook等）で探索的に分析し、結果を管理下に昇格させるという作業パターンが自然に行えること。フレームワークは管理境界IF（I-6）を通じてこのワークフローを支える。

## 構造的整理

要求は6つのカテゴリに分離される。I〜IIIは3つの直交する関心、IVはDAGの進化・合成、Vは辺の設計品質、VIは個人的ワークフロー。

| カテゴリ                       | 問い                         | 要求数            |
| ------------------------------ | ---------------------------- | ----------------- |
| **I. アクセス**                | このデータはどこ？           | 6（I-1〜I-6）     |
| **II. 来歴・整合性**           | このデータはなぜこう？       | 5（II-1〜II-5）   |
| **III. 格納**                  | このデータはどう保持される？ | 3（III-1〜III-3） |
| **IV. 進化と合成**             | DAGはどう変わり、つながる？  | 2（IV-1〜IV-2）   |
| **V. 再利用性**（望ましい）    | 辺の設計品質は？             | 2（V-1〜V-2）     |
| **VI. ワークフロー**（個人的） | どう使うか？                 | 1（VI-1）         |

- 日常のアクセスは意味的属性で行う
- DAGは検証・追跡・再生成の際に参照される
- データを起点として、どの関心にも到達可能
- I-5（異質なデータ構造の共存）はアクセスと格納の両方にまたがる
- I-6（管理境界IF）はIV-2（DAG間合成）の一般化でもある

## スコープの前提

- 利用者は基本的に1人。同時アクセスはスコープ外
- 他者への引き継ぎなどを考慮し、理解可能性・ドキュメント性は必要
- フレームワークの責務境界: 管理下データに対する構造的保証・来歴・再現性・AP/IF提供がコア責務。管理外環境のワークフロー自体は責務外だが、管理境界をまたぐIF提供は責務に含まれる（→I-6）
